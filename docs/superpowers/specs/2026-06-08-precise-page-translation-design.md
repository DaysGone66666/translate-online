# 页面翻译精确定位设计

## 问题

旧实现按最近的块级容器收集文本，并将整个容器的译文追加到容器末尾。这会合并标题、时间、评论数等不同内容，导致译文错位。同时，导航、按钮和短标签被整类跳过，造成明显漏译。

## 方案

页面翻译改为以可见文本节点为最小任务单位：

- 导航、按钮、页眉和页脚文字允许翻译。
- 代码、表单输入、可编辑内容、扩展自身界面和 `translate="no"` 内容继续跳过。
- 短标签允许翻译；纯数字、时间、URL、邮箱和短字母数字标识跳过。
- 短标签和交互控件使用行内译文。
- 段落、标题和较长文本使用下一行译文。
- 译文紧跟源文本节点插入，不再追加到大容器末尾。

## 动态内容

用户启动页面翻译后，使用 `MutationObserver` 监听新增和变化的文本。变化内容经过 300 毫秒防抖后进入增量翻译队列。扩展自身插入的译文和悬浮控件不会触发二次翻译。

取消翻译、页面地址变化或禁用站点时，停止监听、取消请求并移除译文。

## 权衡

文本节点级翻译会产生更多请求，但能显著提升覆盖率和位置准确性。现有翻译缓存继续复用重复文本结果。
